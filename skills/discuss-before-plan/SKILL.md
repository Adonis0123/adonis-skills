# Discuss Before Plan

决策归讨论，计划归 Plan。本 skill 建立一个结构化讨论流程，确保 AI 在进入 Plan Mode 之前，先和用户充分对齐理解、逐一确认关键决策。避免在信息不充分的情况下同时做决策和规划，导致 plan 质量低、反复修改、执行频繁踩坑。

## 输出要求

- **语言**: 中文
- **风格**: 对话式，关键节点输出结构化决策摘要
- **格式**: 讨论过程中自然对话，决策点用结构化格式记录

## 何时使用 vs 何时跳过

### 使用此流程

| 信号 | 示例 |
|------|------|
| 任务涉及多种可行方案 | "给 API 加缓存" — Redis / 内存 / 文件？ |
| 影响面跨多个模块或目录 | "重构认证系统" — 涉及中间件、路由、状态管理 |
| 需求不明确或有歧义 | "让应用更快" — 需要先定位瓶颈 |
| 架构级决策 | "加实时更新" — WebSocket / SSE / 轮询？ |
| 预计触及 5+ 文件 | 新增 API 端点 + 类型 + 测试 + 路由 + UI |
| 用户主动要求讨论 | "我们先聊聊方案" / "帮我分析下选型" |

### 跳过此流程

| 信号 | 示例 |
|------|------|
| 单行或少量行修改 | 修 typo、改常量值 |
| 需求极其明确且无歧义 | "把函数名从 camelCase 改为 snake_case" |
| 用户给出了详细具体的指令 | 已包含文件路径、代码片段、精确行为描述 |
| 用户明确要求跳过讨论 | "直接 plan 吧" / "不用讨论了" |
| 纯执行型任务 | "运行测试" / "格式化代码" |

## 讨论工作流 (SOP)

按 Phase 1 → 2 → 3 → 4 顺序推进。每个 Phase 完成后自然过渡到下一个，不要跳跃。

### Phase 1: 摸底 (Understand)

**目标**: AI 先建立对现状的理解，输出给用户确认，确保双方起点一致。

**做什么**:
1. 读代码、读配置、读文档，分析与任务相关的现状。
2. 输出「我的理解」摘要，包含：
   - 当前系统在任务相关部分的现状
   - 已有的约束条件（技术栈、规范、依赖）
   - 任务的初步理解
3. 标注不确定或不清楚的部分，主动提问。
4. 等用户确认或纠正后再进入下一阶段。

**关键原则**:
- 先调查后发言，不要凭假设输出理解
- 不确定就问，不要猜
- 摸底阶段不做方案推荐

### Phase 2: 讨论 (Discuss)

**目标**: 围绕用户的想法展开技术讨论，充分暴露各方案的优缺点和风险。

**做什么**:
1. 用户抛出想法或方向后，AI 提供：
   - 该方案的优缺点分析
   - 可能的替代方案（至少 1-2 个）
   - 已知风险和潜在坑点
   - 对现有系统的影响评估
2. 如果用户没有明确方向，AI 可以提出 2-3 个候选方案供讨论，但明确标注为「建议」而非「决策」。
3. 讨论过程中主动暴露问题，不隐藏风险、不报喜不报忧。
4. 复杂话题分步讨论，不要一次性抛出所有问题。

**关键原则**:
- 优缺点分析必须平衡，不要只推荐自己偏好的方案
- 风险分析要具体，给出可能的失败场景而非泛泛而谈
- 如果某个方案有致命缺陷，直接指出，不要委婉绕弯

### Phase 3: 决策 (Decide)

**目标**: 将讨论中形成的共识转化为明确的决策记录，逐一和用户确认。

**做什么**:
1. 梳理讨论中已达成的共识，列出决策点清单。
2. 每个决策点包含：
   - 决策问题（一句话描述）
   - 确认的选择
   - 选择理由（一句话）
   - 放弃的替代方案（如有）
3. 逐一和用户确认，未确认的决策不计入最终记录。
4. 如有遗漏的决策点，在此阶段补充讨论。

**决策记录格式**:

```
### 决策记录

| # | 决策问题 | 确认选择 | 理由 |
|---|---------|---------|------|
| 1 | [问题] | [选择] | [理由] |
| 2 | [问题] | [选择] | [理由] |
| ... | ... | ... | ... |

放弃的方案: [简述放弃了什么及原因]
```

**关键原则**:
- 决策记录中只包含用户明确确认的内容
- 有争议或未讨论的点标记为「待定」，不要自行填充
- 决策粒度适中：太粗无法指导执行，太细会拖慢讨论

### Phase 4: 转入 Plan (Transition)

**目标**: 输出决策摘要，建议进入 Plan Mode，由用户最终决定。

**做什么**:
1. 输出完整的决策摘要，包含：
   - 已确认的决策清单（从 Phase 3 整理）
   - 实施范围（预计影响的文件/模块）
   - 已识别的风险及对策
   - 仍待定的事项（如有）
2. 明确建议：「决策已基本完备，建议进入 Plan Mode 做执行步骤分解。」
3. 等用户确认后再进入 Plan Mode。

**决策摘要格式**:

```
## 决策摘要

### 已确认决策
[从决策记录整理的编号列表]

### 实施范围
- [预计影响的文件/模块列表]

### 风险与对策
| 风险 | 对策 |
|------|------|
| [风险描述] | [应对方案] |

### 待定事项
- [如有未决定的事项列在这里]

---
> 决策已基本完备，建议进入 Plan Mode 做执行步骤分解。是否进入？
```

**关键原则**:
- 进入 Plan Mode 的决定权在用户，AI 只建议不执行
- 如果仍有关键决策未确认，主动提醒而非忽略
- 决策摘要是 Plan Mode 的输入，必须足够清晰以支撑步骤分解

## 决策完备性信号

以下信号表明讨论可以收敛，准备转入 Plan Mode：

1. **待讨论问题清空**: 没有标记为「待定」的决策点
2. **收敛信号**: 最近 1-2 轮对话没有产生新的决策点
3. **关键类别已覆盖**:
   - 范围：做什么、不做什么
   - 方案：技术选型、实现路径
   - 约束：性能要求、兼容性、时间限制
   - 接口：对外暴露的 API / 数据格式
   - 风险：已知风险及对策
4. **用户主动表示**: "可以了" / "就这样吧" / "开始 plan"

不需要所有信号同时满足。用户主动表示时即可转入，即使 AI 认为仍有未覆盖的类别——此时可以提醒但不要阻塞。

## 反模式 (Anti-Patterns)

| 编号 | 反模式 | 正确做法 |
|------|--------|---------|
| AP-1 | 未讨论就进 Plan Mode — 跳过讨论直接开始分解执行步骤 | 先走完 Phase 1-3，确认决策后再进 Plan Mode |
| AP-2 | AI 自行做关键决策 — 未经用户确认就选定技术方案或架构 | 所有关键决策必须经用户明确确认，AI 只提供分析和建议 |
| AP-3 | 隐藏已知风险 — 为了推进进度而淡化或省略已知问题 | 主动暴露风险，给出具体的失败场景和影响范围 |
| AP-4 | 一次性抛出所有问题 — 在第一轮对话就列出 10+ 个问题要用户回答 | 分阶段逐步深入，每轮聚焦 2-3 个关键问题 |
| AP-5 | 过度形式化讨论过程 — 每轮都输出完整格式的决策表格 | 讨论中自然对话，只在 Phase 3/4 输出结构化记录 |
| AP-6 | 决策摘要中引入未确认内容 — 把 AI 的建议写成已确认决策 | 决策摘要严格只包含用户明确 confirm 的内容 |
| AP-7 | Plan Mode 中偷偷做新决策 — 在步骤分解时引入讨论阶段未提及的方案选择 | Plan Mode 只做执行步骤分解，新决策必须退回讨论阶段 |

## 各阶段提示模板

### Phase 1 摸底阶段

- "让我先看一下现有代码和相关配置，了解下当前的情况。"
- "以下是我对当前系统的理解，请确认是否准确：[摘要]。有什么需要纠正的吗？"
- "关于 [某部分] 我不太确定，能否说明一下 [具体问题]？"
- "现状确认完毕，接下来我们讨论方案。你有倾向的方向吗？"

### Phase 2 讨论阶段

- "你提到的 [方案 A] 有以下优缺点：[分析]。另外还有 [方案 B] 可以考虑，主要区别在于 [差异]。"
- "这个方案有一个潜在风险需要注意：[具体风险]。可能的影响是 [场景]，建议的对策是 [方案]。"
- "目前有两个关键问题需要先确定：1. [问题一]；2. [问题二]。先聊哪个？"
- "这个点我建议选 [选项]，原因是 [理由]。不过最终由你决定。"

### Phase 3 决策阶段

- "根据刚才的讨论，我整理了以下决策点，请逐一确认：[决策列表]"
- "还有一个之前没讨论到的点：[遗漏项]。你怎么看？"
- "所有决策已确认，没有待定事项。准备整理决策摘要。"

### Phase 4 转入阶段

- "决策摘要如上。所有关键决策已确认，建议进入 Plan Mode 做执行步骤分解。是否进入？"
- "注意：[待定事项] 尚未确定，可以先进 Plan Mode 处理已确定的部分，也可以先把这个问题讨论清楚。你怎么选？"
