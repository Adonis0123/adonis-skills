# 治理关注点 — 五件套之外的四件事

五件套解决的是**扩展成本 O(1)**。但当插件机制要长期运营（>1 年）、跨团队共用、或对外开放第三方插件时，会撞上四个五件套不覆盖的治理问题。业界同类指南几乎都不讲测试策略——这一节是补上这块盲区。

**适用判断**：内部小团队、插件作者=Core 维护者、生命周期短的机制,可以整篇跳过——治理是规模和时间带来的问题,不要提前建设（YAGNI）。

## 一、Contract 版本演进

Contract 改动是全体插件的同步成本（SKILL.md Contract 小节已讲"加字段优先用可选"）。当"可选字段"不够用时的升级路径,按代价从低到高：

1. **可选字段 + 默认值兜底**——绝大多数演进停在这层就够。
2. **能力探测**：Core 检查插件是否实现了某可选方法再调用（如 Vite 插件靠 `this.meta.rolldownVersion` 探测宿主能力,反向同理）。
3. **显式版本字段**：Contract 加 `apiVersion`,Core 对旧版本走 adapter（Microkernel 模式里 registry 记录契约版本的用意）。
4. **平行 Contract**：`ContractV2` 与 V1 并存 + 迁移期,最后删 V1——代价最高,只在字段语义发生根本变化时用。

**反模式**：直接改必填字段语义而不改名不加版本——所有插件静默错行为,比编译错误糟糕得多。

## 二、隔离与权限

当插件作者不再等于 Core 维护者（第三方/跨团队）,"契约"之外还需要"权限"：

- **能力注入而非全局访问**：插件只拿到 Core 显式传入的依赖（Tailwind 的 `({ addUtilities, theme })`、Tauri v2 的命名 permission + capabilities 授权）,不允许插件自己 import Core 内部模块。
- **声明式授权**：插件声明需要什么能力,宿主决定给不给（Tauri v2 `src-tauri/capabilities/` 是目前最完整的参照）。
- **故障爆炸半径**：一个插件抛错不能拖垮 Core 和其他插件——见下一节。

内部机制可以只做第一条（能力注入是好设计,不只是安全需求）。

## 三、生命周期失败处理

声明式注册的隐含承诺是"注册了就能用"。真实运行时要回答三个问题：

1. **加载失败**：延迟加载的 chunk 拉不下来 / 插件 init 抛错——Core 要有兜底（跳过该插件 + 上报,而不是整个 Provider 崩溃）。React 场景给插件渲染层包 ErrorBoundary 是最低标准。
2. **禁用通道**：`disabled` 字段（编译期）之外,长期运营的机制通常还需要运行时开关（远程配置杀开关）,用于"线上某插件出事故,不发版下线"。
3. **降级顺序**：编排层选中的插件失败后,是选下一个 priority 还是什么都不显示？——这个决策要写进 Orchestrator,不能靠默认行为碰运气。

## 四、插件测试策略

插件化机制的测试分三层,**大多数团队只写了第一层**：

1. **单插件测试**：每个 Convention Folder 内测自己的 `init.ts` 条件逻辑和 UI——普通组件测试,无特殊之处。
2. **Contract 合规测试（最值钱、最少人写）**：一套**参数化的契约测试**跑在*每一个*注册表条目上——"所有插件的必填字段齐全"、"所有 key 符合命名推导表"、"所有 component 可被 lazy 加载成功"。写一次,每加一个插件自动多一组用例,这正是插件化在测试维度的 O(1)。评审清单里 grep 类检查（Core 无具体 key、无 cache 双写）挂进 lint/CI 也属于这层。
3. **机制测试**：Core 状态机 + 编排层用**假插件**（test fixture 注册表）测试——缓存策略、priority 竞争、force 重算、维度隔离。用假插件而不是真插件,机制测试才不会因业务插件增删而抖动。

**落地最小集**：第 2 层的参数化契约测试 + 第 3 层的假插件机制测试。第 1 层按各插件业务风险自行决定。
