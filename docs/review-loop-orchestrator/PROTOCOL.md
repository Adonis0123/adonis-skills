# 接力模式协议（Relay Mode）

> 协作席：**你**（用户 / Boss）+ AI 窗口。默认双席 **@opus** + **@codex**；可选第三席 **@grok**（Grok Build / 本机 `~/.grok`）。
> 每条协作线 = 一个按主题命名的接力文件，**用文件名寻址，无共享指针**。
> 用户只当"接力棒"——一句话喊醒某一方；review/改动的**内容**由各方自己去接力文件读，用户从不复制粘贴转发。

### 席位与默认分工（2026-07-14 · @grok 入席）

| 署名       | 典型窗口                                    | 默认分工                                                                                     |
| ---------- | ------------------------------------------- | -------------------------------------------------------------------------------------------- |
| **@opus**  | Claude / Opus 交互或通道                    | 设计推进、实现、与 Codex 双盲/接力主循环                                                     |
| **@codex** | Codex                                       | 实现、严格 review、状态机/合同硬化                                                           |
| **@grok**  | Grok Build（skill: `~/.grok/skills/relay`） | **第三人意见**：架构吐槽、交付/E2E 视角、防自我合理化；默认不当主 doer（baton 另有指定除外） |
| **@me**    | 用户                                        | 出题、插话、定稿 / 收工                                                                      |

- 双席日常开发：仍 **@opus ↔ @codex**，无需叫 @grok。
- 需要外部挑剔 / 成片交付闭环 / 「别自己夸自己」时：开线并 baton **@grok** 先写立场，再交给 opus/codex 对轰或收口。
- **自动直连讨论**（`coordinator-lite.sh`）当前仍是 **opus|codex 双模型**（代码强制）。三席辩论用 **人肉交棒** 即可；不要为 100 分去先改 coordinator 成三方自动轮转（95 分纪律）。

## 设计纪律：够用即止，不为 95→100 过度设计（总纲）

**红线：不允许 bug。** 正确性、契约、安全、数据丢失——必须堵死，没得商量。

**但达到「没 bug + 能用 + ~95 分」之后，为把它推到 100 分而做一个很重的设计/抽象/新层/新流程，是错的。** 那 5 分的收益配不上重设计带来的复杂度、维护成本和新出 bug 的风险。

动手前问一句：这个设计是**堵一个真 bug / 真风险**，还是**为了更优雅 / 更完备 / 覆盖一个几乎不发生的边角**？前者做，后者砍成 backlog 或直接不做。

- 对 doer：先交 95 分的简单解，别一上来盖满分的大楼。
- 对 reviewer：不能拿「还能更完美」拦路（见收敛规则），更不能主动提议重设计去追满分。

对应 CLAUDE.md §2 简单优先、review 高 ROI。**这条优先级高于本协议其他所有「更完善」的诉求。**

## 激活速查（方案已接入各端）

各端都已内置"如何开启接力"，用户只说一句即可：

- **Opus 窗口**：说 `进入接力` → 靠 Opus memory 命中。
- **Codex 窗口**：说 `进入接力` → 靠 Codex `relay` skill（`~/.codex/skills/relay/SKILL.md`）命中。
- **Grok 窗口**：说 `进入接力` → 靠 Grok skill `~/.grok/skills/relay/SKILL.md` 命中；署名默认 **@grok**。

并行多线只靠**不同的命名文件**区分（baton 点名哪个就是哪条），**不再用线路名/room/指针**。新开的独立窗口不说这句就完全不受影响。

## 硬前提

**参与接力的所有窗口必须在同一 repo checkout 根目录、同一文件系统上启动。** 窗口若在别的项目目录，Codex 沙箱 / 相对路径都够不到 `tmp/relay/`，接力不成立。这是地基，先确认再干活。

**worktree 例外（接力文件永远走主库绝对路径）**：并行线用各自 worktree 隔离**代码**是对的，但 `tmp/relay/` 只存在于主库、且 gitignored——**worktree 里的 `tmp/relay/` 是空的**。所以接力文件永远用**主库绝对路径**读写（`/主库根/tmp/relay/...`），或先回到主库根再操作。别在 worktree 的相对 `tmp/relay/` 上找文件，那里啥都没有。

## 参与者与署名

默认 AI 署名为 **@opus**、**@codex**；第三席为 **@grok**。如果某个模型/窗口不可用，用户可以开另一个 AI 窗口接棒；替补窗口必须在第一条接力消息里声明一个**唯一署名**，例如 `@codex2`、`@codex-reviewer`、`@opus2`、`@grok2`。

同一接力文件内，同时活跃的窗口**不能共用同一个署名**。署名是判断"对方最新块"和"我自己的上一条 review"的依据；如果两个 Codex 都写 `@codex`，复审和交棒会混线。用户可用类似这句唤起替补窗口：

```text
接力 · 你作为 @codex2 接棒，先读 PROTOCOL 和 tmp/relay/<本线文件>.md 的 tail，继续 review
```

唤起 Grok 第三人意见示例：

```text
接力 · 读 tmp/relay/<本线文件>.md · 你是 @grok，先读 tail，就成片/交付问题给出挑剔立场（先证据后观点），末行交棒给 @codex 或 @opus
```

## 评审加固（v2 硬规则）

第一轮 Codex 评审后固化的载重规则：

1. **交棒行完整复制**：交棒指令必须原样复制粘贴，**不改写、不省略**（尤其里面点名的文件路径别丢）；万一手打，也必须以 `接力` 开头 + 带文件路径。这是接收方进入 relay 并锚回正确线的唯一可靠入口。
2. **读在先（但别重复读全量，省 token）**：**首次**进接力才读全 `PROTOCOL.md`；此后每轮只读 **baton 点名的接力文件 tail（最后一两块）**再动手——不要每轮重读整份协议、也不要重读越滚越大的整个接力文件。只有会话被压缩、协议已不在上下文时，才重读一次协议。这样既保确定性又省 token。
3. **diff 基准**：第二轮 review 起、或 diff 明显变大 / 涉及共享模块 → doer 本轮 commit，done 块写 `base=HEAD~1` + 改了哪些文件；reviewer 用 `git diff HEAD~1`（或声明的 base）。小的第一轮不必强制 commit。
4. **并发写**：append 前重读 tail、append 后确认自己的块落上了；两窗口尽量不同时写同一文件（靠轮流）。
5. **并行隔离（默认文件分区，别默认 worktree）**：并行多线改代码**默认走文件分区**——每条线只碰**不同文件/目录**，配提交纪律（只 `git add` 自己的文件），reviewer 只看 `git diff -- <paths>`。**不默认开 worktree**：pnpm monorepo 里每个 worktree 要单独装 node_modules，很烦。只有**两条线真要改同一批文件**、文件分区做不到时才 worktree（那时 warm-store `pnpm install` 主要是链接非下载，或 symlink 复用主库 node_modules）。

## 崩溃恢复与状态可见（v3：实战补丁）

窗口会死（API 502、崩溃、被关）。协议不能默认两窗口都活着。两条最小补丁（本身也守总纲，别做重）：

1. **状态头**：每个接力文件标题下、第一个消息块前，维护**一行** STATE，谁写谁顺手更新——一眼看清进度，不用读整个 tail 脑补「球在谁手里」。它是「现在到哪了」的唯一权威。
   ```
   STATE: 批次=<当前批> | 球=@<谁的回合> | 阻塞=<无/具体项> | 已定稿=<列表>
   ```
2. **重连剧本**（新窗口顶替掉线的一方）：读 `PROTOCOL.md`（首次）+ baton 点名的接力文件的 顶部 STATE + tail 最后一两块 → 用**原署名**接管（顶替 @opus 就还叫 @opus；若原 @opus 仍活着才用 @opus2）→ 掉线前那半截活儿从 STATE + tail 判断做完没，没做完就**从头重做该回合**，别猜「上一个我」的隐藏进度。**掉线是基础设施问题、不是流程回退**：恢复后按 STATE 继续，不重头 battle 已定稿的批次。

## 证据纪律：一致认为要查，就真去查

**当两窗口一致认为某一点需要查资料/证据（而不是各执假设互相说服），就实际去查，再基于查到的东西讨论——不靠记忆或直觉下结论。** 查的来源按成本从低到高：先翻**已有实证记忆/文档**（很多坑已经踩过、别重复），不够再 web/官方文档/最小实测 spike。

- 尤其成片：设计新字段/规则/控制位前，核 Seedance 真实能力与硬限制（见 [[feedback_video_verify_seedance_capability_before_design]]）。
- 查完把**结论 + 来源**写进接力块，让对方能复核，不是口头断言。
- 这条守总纲：查是为了定一个真决策，不是为完美无限调研；已有证据够用就停。

## 收敛与终止（防暴君、防没完没了）

AI-AI 评审天然会往"暴君"（死不放行）和"没完没了"（一轮轮加戏）两个方向滑。以下是硬约束：

1. **用户是最终权威**：用户说 `定稿` / `收工` / `ship` / `就这样` → **立即结束，任何 AI 不得再拦**。AI 至多在收尾附一句风险提示，但**不能否决用户的发布决定**。用户也可以随时 `叫停` / `换方向` / `这条不改了`。
2. **拦路必须可证伪**：reviewer 想 block，必须点名一个具体的"这样会坏在 X"（正确性/契约/安全/数据丢失/承重 guard）。**给不出具体 breakage 就必须放行**。"感觉能更好 / 不够优雅 / 风格 / 口味"**永远不是阻塞理由**。
3. **阻塞 vs 非阻塞**：每条 review 意见标 `[阻塞]` 或 `[非阻塞]`。只有 `[阻塞]` 能拦定稿；`[非阻塞]` 一律记进 backlog，不挡路。
4. **不许加戏**：不能一轮轮新造非阻塞小意见把接力吊着。第二轮起，新意见只有**真 load-bearing 的阻塞项**才允许提出；否则本轮就该收敛。
5. **轮次预算**：默认单件事 **≤3 轮**。到上限若只剩 `[非阻塞]` → 自动收敛为"定稿 + 剩余记 backlog"。
6. **死锁上交用户**：两个 AI 真对不齐（一个坚持阻塞、一个不认）→ **不许继续对轰**。一句话把分歧+各自理由+推荐摆给用户，用户拍板（对应"surface conflicts, don't average them"）。

一句话：**能点名具体 breakage 才配拦路；拦不动就放行或上交你，绝不空转。**

## 文件布局

- **唯一寻址方式：baton 点名文件路径。** 交棒行必须含具体 `tmp/relay/<…>.md` 路径——那个文件就是本次的线，直接按它读写。**没有 CURRENT 指针、没有"当前线"这种共享可变状态**（它并行下会被别的窗口抢写，是"串线"的根源，已废弃）。
- `tmp/relay/<时间>-<主题>.md` —— 每条协作线一个新文件，**append-only**（只往末尾追加）。**文件名（主题）就是这条线的稳定 ID**，用它寻址，不用指针。旧文件即历史归档，别删。
- **本窗口私有记忆定位**：某条 baton 点名文件后，本窗口本会话就 working 在那个文件上；后续 bare `接力 · …`（没点新文件）默认**继续同一个文件**；baton 点名另一个文件即切线。这是**每个窗口私有的记忆，不是共享指针**，并行永不撞车。
- `tmp/relay/new-relay.sh "<主题>"` —— 开新线：建文件并**打印其路径**。用户把该路径放进 baton 发给两边。

**每次协作开一个新接力文件**，不要把所有话题堆进一个文件。

## 并行多条接力线（用文件名，不用指针/room）

并行多线 = **多个命名文件，各自独立**。baton 点哪个文件就是哪条线，天然不撞车——**不需要 room / 线路名 / `CURRENT.<线路>` 指针**（那些也是会被抢写的共享状态，同样废弃）。

- 起新并行线：`new-relay.sh "<主题>"` 建文件，把打印出的路径放进 baton。
- 每个窗口本会话**记住自己在哪个文件上**（私有记忆，不是共享指针）；切线靠 baton 点名新文件。
- 交棒行格式 `接力 · 读 tmp/relay/<文件>.md · <指令>`：`接力` 是 relay 入口**关键词**，**文件路径是这条线的唯一锚**。即使会话被压缩、窗口忘了自己在接力，看到 `接力` 会重新命中 relay 模式，靠路径锚回正确的线。

## 代码增量与隔离（reviewer 看的是真 diff）

所有窗口共享同一份 git 工作区。`git diff` 是"改了什么"的真相，但要注意它是**累积**的。按需分级：

- **提交纪律（共享工作区，2026-07-03 实际踩过）**：commit / amend 前先 `git status` 看全树；**只显式 `git add <自己的文件>`，绝不 `git add -A` / `git commit -a`**；别人的 WIP 在树里就别扫进来。**慎用 `--amend`**——它会把 index 里别人已 stage 的文件一起卷进你的提交（当天就这么把 @codex 的 WIP 卷进了 @opus 的提交、还带错了署名）。不确定 HEAD 是不是自己的、index 里有没有别人的东西，就别 amend，新开一条 commit。
- **热点共享文件（文件分区对它们无效）**：少数文件多条线都要动，分区救不了——典型是 `video-production-craft-library.ts`、schema 源 → 重生成的 70 个 generated tool（源分了、生成物仍撞）。规矩：**同一时刻只一条线动热点文件**，另一条等或先干别处；**真撞了用 `git add -p` 只 stage 自己的 hunk**，绝不整文件 `git add`。提交前 `git status` 看到热点文件里有别人的改动 → 停下核清楚，别把它一起提。
- **别回退别人的改动（discard 纪律）**：树是共享的。**禁止树级破坏命令**——`git reset --hard` / `git checkout .` / `git restore .` / `git stash` / `git clean -f` 会连别人未提交的活一起抹掉。要丢弃只丢**自己那一个文件**，且丢之前 `git diff <file>` 确认里面只有你的改动。改共享文件**优先 Edit（定点替换），别用整文件 Write**（Write 会把文件里别人的改动一起覆盖回退）；工具报"文件已被改动"就**重读、看清别人的改动再动，别强推/别绕过**。**自己的活尽早小步 commit**——提交后别人的 working-tree 操作就删不掉你的了。revert/reset 某个 commit 前先 `git show` 确认它整条都是你的，**绝不 revert/reset 混了多线的 commit**。
- **单线随手迭代**：diff 还小，reviewer 直接看整份 `git diff` 即可。doer 在 done 块里写清**本轮动了哪些文件**，帮 reviewer 定位增量。
- **单线多轮 / diff 变大**：doer **每轮改完 commit 一次**（可用临时分支），reviewer 就能 `git diff HEAD~1` 精确看**这一轮**的增量，不被历史累积干扰。
- **并行多线（默认文件分区）**：两条线都改代码时 diff 会混在一起。**默认解法=文件分区**：约定两条线只碰**不同文件/目录**，配提交纪律（只 add 自己的文件），reviewer 只看 `git diff -- <本线 paths>`。**worktree 不是默认**——pnpm monorepo 每个 worktree 要单独装 node_modules 很烦；只有两条线真要改同一批文件、分区做不到时才用 `EnterWorktree`，那时靠 warm-store `pnpm install`（链接为主）或 symlink 复用主库 node_modules 省掉重装。

## 每条消息块格式

```
## @opus · MM-DD HH:MM
（内容：review 结论 / 改了什么 / done / 决定 …）
```

署名：默认 Claude/Opus → `@opus`；默认 Codex → `@codex`；默认 Grok → `@grok`；用户 → `@me`。如果用户声明了替补身份（如 `@codex2` / `@grok2`），以声明身份为准，并在本接力文件后续块里保持一致。时间戳可选。

**done 块轻模板**（只对"改完"的块，别把每个块都套死）：末尾带一行 `changed: <文件列表>`，需要精确基准时再加 `base: HEAD~1`。让 reviewer 只读 tail 也能定位增量。

## 开新接力

用户说"开新接力：<主题>"时，听到的窗口跑：

```bash
tmp/relay/new-relay.sh "<主题>"          # 建文件并打印其路径
```

它新建接力文件并**打印文件路径**。用户把该路径放进 baton（`接力 · 读 tmp/relay/<该文件>.md · …`）发给两边，两边就锚到同一条线——**靠文件名，不靠指针**。

## 进入接力模式（本窗口的职责）

一旦本窗口读了本协议，就进入接力模式。此后每次用户给你指令时：

1. **定位文件**：**baton 点名的文件路径就是这条线**——直接按它读写，静默执行。若 baton 没点名文件（bare `接力 · …`），用**本窗口私有记忆**里正 working 的那个文件（上一条点名过的）继续。两者都没有（全新窗口 + 无文件）→ 提示"baton 里点名文件路径"或按主题跑 `new-relay.sh` 建一条。**永不读 CURRENT 指针（已废弃）。**
2. **读自上次发言以来的所有新增块**：不是只读"最新一块"，而是读**你自己上一条块之后追加的全部块**——既包含对方（reviewer/doer）的块，**也包含用户 `@me` 中途插入的需求/变更**。这样你既知道"对方刚干了什么"，也不会漏掉用户插进来的方案改动。若这些新块里有 `@me` 的需求变更，**优先按它调整**，别按旧方案埋头跑。
3. **按指令干活**：
   - 让你 **review** → 读对方的实际改动（`git diff`，见下方"代码增量与隔离"）+ 接力文件里用户意图 → 把 review 结论**追加**成你的块。**不要**等用户把改动贴给你。
     - **复审（第二轮及以后）**：先回看**你自己上一条 review 块**，对照本轮 diff **逐条**确认之前的意见是否已处理，再写新结论。别只看对方的 done 块就放行。
   - 让你 **按 review 改 / 去改** → 读接力文件里最新的 review 块 → 改代码 → 追加一个简短的 `done + 改了啥` 块。**不要**等用户把 review 贴给你。
4. **只读不搬运**：对方的内容你自己去接力文件和 git 里读，永远不需要用户帮你转发大段信息。
5. **交棒输出（重要）**：干完并写入接力文件后，你给用户的回复**结尾必须单独输出一行"交棒指令"**——一句用户能**原样复制**粘贴到对方窗口的短话。用户不用记任何 ID。格式：

   ```
   ▶ 交棒 · 复制到对方窗口：接力 · 读 tmp/relay/<本线文件>.md · <短指令>
   ```

   例：
   - `▶ 交棒 · 复制到 codex 窗口：接力 · 读 tmp/relay/20260703-1658-对白丢失-P0.md · 去改（见文件里我的 review）`
   - `▶ 交棒 · 复制到 opus 窗口：接力 · 读 tmp/relay/20260703-1658-对白丢失-P0.md · 继续 review`

   **必须以 `接力` 开头 + 点名文件路径**：`接力` 是**关键词**（对方窗口就算 context 被压缩、忘了自己在接力，看到它也会重新命中接力模式），**文件路径是唯一的线锚**（锚回正确的线，不靠任何指针）。两者都别省。

## 触发词（宽松匹配，用户会用短句）

进入接力模式后，短句即可，按角色理解意图：

- 开新接力：`开新接力：<主题>` / `新接力 <主题>`
- 让人 review：`review` / `继续 review` / `再看看`
- 让人改：`去改` / `改一下` / `看 relay` / `codex 上`
- 附带指针：`顺便看下 xxx 文档` → 一并读那份文档

## 入口分诊（用户只管抛想法，窗口判断归属 + 建议时机）

用户常判断不准"这件事属不属于当前接力、该现在做还是以后做、是不是你们能做的"。所以：**用户可以把任何想法/需求直接抛给任意接力窗口，不必自己预先分类。** 收到后窗口**先分诊、再动手**，用一句话告诉用户归属和建议时机：

- **属于当前任务** → 直接纳入，继续。
- **相关但独立的小事** → 顺手做，或记进当前文件 backlog。
- **独立任务、我们能做** → 给安排建议：
  - 跟当前不冲突 → 建议开**并行线**（给出交棒行），可现在并行；
  - 依赖当前结果 / 会抢同一份代码 → 建议**排在当前之后**；
  - 不急 → 记 **backlog**，回头做。
- **不属于我们 / 需用户拍板或用别的工具** → 直说，别硬接。

分诊是判断题（该由模型做），不是让用户来分。用户永远可否决建议、直接指定怎么安排。

### 自主切/开新线（明显不是一条线时，别静默、别过碎）

当**明显不是当前这条线**——用户抛来一个完全不同的主题/人物/子系统，或窗口自己判断"这不属于当前线"——**主动开/切一条新线，不要往当前线里硬塞，也不要只是含糊建议**。

- **判据=是不是同一道题**：同一决策、同一批文件、会被一起 review → 留本线；**不同子系统/不同决策/不会一起 review → 开新线**。同一道题的子问题（如某设计的一个环节）留本线，别切。
- **自主 ≠ 静默**：一旦切/开新线，**必须显式告诉用户新线文件名 + 在旧线留一句移交指针**，让用户知道 baton 往哪条递、让对方窗口收敛到同一条线（否则又变多线并发撞车）。
- **别过度碎片化**：只为**真正独立**的题开新线；不确定是否独立就倾向留本线。碎成一堆线，用户反而找不着。
- 用户可随时否决、指定合并或留在原线。

## 会话快照（handoff，可选加速器）

会话收尾且多线在飞时，可生成一份跨线快照 `tmp/handoff-YYYYMMDD-<主题>.md`（Claude 用 /handoff 生成后拷入；结构：通用节对所有 agent + 各 agent 附录）。规则：

- **零独占承重事实**：任何承重信息（球位/拍板/路径/坑）必须同时存在于 relay 或长期记忆；快照只装冗余的方便。满足此条快照才可安全丢弃/过时。
- **快照语义**：生成后永不更新；要新的重新生成；relay 才是活文档。信任顺序 relay > 记忆 > 快照。
- **接棒句式**：`接力 · [先读 tmp/handoff-xxx.md] · 读 tmp/relay/<账本>.md 以 STATE 为准 · <任务>`。快照可选，漏带不出错只会慢。
- 单线短会话不需要快照——relay 的 STATE + 末块就是天然 handoff。

## 会话地址（署名带地址，可选）

每个 Claude 窗口有稳定会话 ID（transcript 在 `~/.claude/projects/<项目>/<sessionId>.jsonl`）。窗口可在进接力的首块登记 `sessionAddr: <id>`。对方（Codex/orchestrator/脚本）要"找这个窗口"时：

```bash
claude --resume <sessionId> -p "接力 · 读 tmp/relay/<账本>.md · <任务>"
```

语义与纪律：

- **唤醒的是该会话的完整上下文副本（分叉），不是正开着的屏幕**——原窗口无感知，两条时间线可能各自演化。因此被唤醒方产出**必须写回 relay** 才算存在；账本只有一本，分叉只是草稿纸。
- 地址是便利不是承重（同快照零独占铁律）：resume 失败就按普通接棒拉新窗口，靠记忆+relay+快照恢复。
- 大目标模式下 orchestrator 优先按地址唤醒（继承会话内上下文），拉全新窗口作 fallback。

## 直连讨论模式（用户只发令，不传话）

用户对任一窗口说"去找 opus/codex 讨论 X"即触发。被指示方在**自己的当前回合内**同步驱动整场讨论，用户不再复制粘贴：

1. 查 `tmp/relay/ADDRESSES.md` 拿对方通道地址；缺失/失效则退回人肉接棒并告知用户。
2. 先把自己的立场写成 relay 块（账本永远先行）。
3. 以阻塞子进程唤醒对方（命令见 ADDRESSES.md），接力行点名账本文件；对方在**自己的同一会话**里处理并写回。
4. 读账本 tail 取回应，**每次用户指令最多 3 个来回**，然后向用户汇总并停下——绝不无人值守续跑。
5. 硬边界：直连回合只做讨论/review/裁决并写回 relay；**双方都不得**在直连回合里实施/发布/push/写 DB/动实例。
6. 轮换自觉：任一方感到会话退化（重度压缩/丢线索）→ 在 relay 写"建议轮换"块 + 生成 handoff → 新会话登记新地址进 ADDRESSES.md，旧行标 retired。
7. 成本模型：投递（写文件）零成本；唤醒对方=对方付一次自己上下文的钱；无守望进程、无空轮询。
